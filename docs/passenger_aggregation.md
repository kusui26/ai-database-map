# 駅別乗降客数（S12-25）集約設計

国土数値情報「駅別乗降客数」(S12-25) を、**「同じ駅名・半径1km以内」で集約**し、AI Database Map で使う指標（各年の乗降客数の推移・増減率）を生成するための設計仕様。データ整形は `script/create_dataset_for_AI_Database_Map.ipynb` で行う。CRS の扱いは [`CRS.md`](./CRS.md) を参照。

本書は**結論だけでなく、結論に至った検証プロセス・不採用にした案・残存リスク**も記録する。すべての数値は `data/S12-25_GML.zip`（UTF-8版 GeoJSON, 10,534行）の実測に基づく。

## 1. サマリー

| 項目 | 決定 |
|---|---|
| 空間集約 | **駅名 + 半径1km**（代表点→駅名ごとに haversine DBSCAN, eps=1000m）|
| 構成単位 | **運営会社**（路線は事業者内で合算）|
| 在/欠測の判定 | **データ有無コード==1** を実測（値0も実測の0）、それ以外は欠測 |
| レベル時系列 | *inclusive*（在の全社合算）・**補間なし**・全社欠測年はNULL・寄与社数で透明化 |
| 増減率 | *restrictive*（比較2年で揃う社だけの like-for-like basket）|
| 出力 | **2テーブル**：メイン `station_group`（1群1行・ワイド）＋ 監査用 `station_group_operator`（1群×1社）|

**設計の背骨**：
- **レベルは「全部足す」、比率は「揃う社だけ」** ── 目的が違うので合算ロジックを分ける。
- **欠測(データ有無≠1)を0として扱わない** ── 見かけの急増減を防ぐ最重要原則。
- **比率は合算から導出できない**（basketが違う）ので、両方を別列で保持する。

## 2. データ構造の解読

メタデータ(KS-META)にコードリスト定義が無かったため、コードの意味は**実データから経験的に確定**した。

### 2.1 列構成
- 固定: `S12_001`駅名, `S12_001c`駅コード, `S12_001g`グループコード, `S12_002`運営会社, `S12_003`路線名, `S12_004`鉄道区分, `S12_005`事業者種別
- 年度別（2011–2024 の14年 × 4列）: `重複コード` / `データ有無コード` / `備考` / `乗降客数`
  - 2011=`S12_006`〜`S12_009` … 2024=`S12_058`〜`S12_061`
- 形状: `LINESTRING`（駅の線路区間）、CRS = **EPSG:6668 (JGD2011)**

### 2.2 重複コード（年ごと）── 二重計上の防止
複数路線・複数事業者の駅で、データを一箇所に集約するための仕組み。

| 値 | 意味 | 乗降客数 |
|---|---|---|
| `1` | **代表行**（その事業者の集約先）| 値を持つ |
| `2` | **重複行**（他行に集約済み）| **必ず0** |
| `3` | 対象外（未開業・廃止等の重複）| 0 |

> **検証**：東京駅のJR東日本は8行あるが、東海道線の1行（重複=1, 869,128）だけに値があり、京葉/総武/東北新幹線/東北線（重複=2）は全て0。**事業者内で複数路線が1行に合算**される。

### 2.3 データ有無コード（年ごと）── 最重要
| 値 | 意味 |
|---|---|
| `1` | **実測あり**（値が0でも「実測された0」。年800件ほど存在）|
| `2` / `3` / `4` | **欠測**（その年に値なし。未開業・非公開・対象外）|

> ⚠️ **欠測の符号は2019年に `3`→`2` へ変更**された（2011-2018は主に3, 2019以降は2）。
> → 「`==1` を実測、それ以外を欠測」と判定するのが唯一の堅牢な方法。`3`決め打ちは破綻する。

### 2.4 「0」には2種類ある（核心）
- **実測の0**（データ有無=1, 値0）── 小規模駅など。**採用**。
- **欠測**（データ有無≠1）── データが無い。**0扱い厳禁**（合算・比較から除外）。

欠測を0として扱うと、路線の計上開始/終了で**見かけの急増減**が発生する。これが全設計の出発点。

## 3. 集約の基本単位

### 3.1 空間グループ：駅名 + 半径1km
- 各行の代表点（`geometry.representative_point()`）を取り、**駅名ごとに** haversine DBSCAN（eps = 1000m / 地球半径, min_samples=1）でクラスタリング。
- 結果: **9,273群**（10,534行から）。内訳: 単独行 8,401群、複数行 872群。
- 同名でも1km超離れていれば別群になる（**406駅名**が複数群に分離。例: 市役所前, 県庁前, 高田）。`grp` キーは `"駅名#クラスタ番号"`（例: `東京#0`）で駅名を内包する。

> **なぜ駅名だけでなく空間も使うか**：「県庁前」等の同名駅は全国に多数あり、駅名のみの集約では別都市の駅が混ざる。1km制約がこれを防ぐ。
> **なぜ既存の `S12_001g`（グループコード）を使わないか**：あれは KSJ の「300m + 同名」グループで、ユーザー要件の1kmより狭い（例: 東京駅の京葉線は別グループになる）。要件に合わせ独自に1kmで作る。

### 3.2 構成単位：運営会社
路線単位ではなく**運営会社単位**で集約する。
- 理由: 事業者内では路線が重複コードにより1行へ合算され、かつ**代表行が年によって移動する**（路線単位は不安定）。事業者単位なら値が保存される。
- 実装: `(grp, 運営会社, 年)` で乗降客数を `sum`（重複行は0なので二重計上なし）、`データ有無==1` の有無を `any` で集計。

### 3.3 在/不在の判定
- **在（present）= その年に `データ有無コード==1` の行が1つでもある**（値0でも実測として在）。
- **欠測 = それ以外**（NaN扱い）。

## 4. 指標①：レベル時系列（2011–2024の推移）

グラフ表示用。「その年に実際に何人使ったか」を見せるので **inclusive（在の全社合算）**。

### 4.1 ロジック（補間なし）
1. 運営会社内で路線を合算（重複コードで二重計上なし）。
2. 在=`データ有無==1`（値0も実測）、欠測=NaN。
3. **補間しない**。各年、在の社をそのまま合算（`sum(min_count=1)`）。全社欠測の年だけNaN（グラフは線を切る）。
4. **寄与社数 `n_op` を併記**（その年に実測した社数）。社が抜けた年は `n_op` が下がり、「谷」がデータ穴か実減かを判別できる。

### 4.2 実態：問題は局所的
| 観点 | 結果 |
|---|---|
| 在の社集合が**全年一定**（段差リスクなし）| **9,109群 / 98.2%** |
| 構成変動（途中で社が出入り）| 164群 / 1.8% |
| 内部欠損（在に挟まれた欠測）を持つ群 | 480群 |

主要駅（東京・新宿・渋谷）は内部欠損ゼロ。欠損は北越急行・JR地方線など**小規模単独路線に集中**。

### 4.3 例：東京駅（補間なし）
```
       2011   2012   2013    2014    2015    2016    2017    2018    2019   2020   2021    2022    2023    2024
pax    920k   931k   986k   1018k   1033k   1074k   1091k   1153k  1340k   732k   787k   1006k   1177k   1263k
n_op      2      2      2       2       2       2       2       2      3      3      3       3       3       3
```
- 右肩上がり → コロナ急落(2020) → 回復、という実態が見える。
- 2019に `n_op` が 2→3（JR東海・東海道新幹線が計上開始）。2018→2019 の +16% には新規社分が混入＝**段差**。`n_op` で透明化され、UIで注記できる。この駅の公正な増減率は比率列（§6, −5.7%）が別途持つ。

### 4.4 補間しない理由 → §9.5（コロナ年の過大評価懸念）

## 5. 指標②：増減率パターン1（直近 2024/2023）

### 5.1 ロジック
- 群ごとに、**2023と2024の両方で実測があり、かつ2023値>0** の運営会社を basket とする。
- `rate_yoy = Σ2024(basket) / Σ2023(basket) − 1`。

### 5.2 実測
- 算出可能: **7,367群（79.4%）**、中央値 **+2.14%**（四分位 −0.7%〜+5.1%）。コロナからの回復基調で妥当。

| 例 | 2023 | 2024 | rate_yoy | basket |
|---|---|---|---|---|
| 東京 | 1,176,902 | 1,262,604 | +7.3% | 3社 |
| 新宿 | 2,984,711 | 2,713,386 | −9.1% | 4社 |
| 渋谷 | 2,129,874 | 2,897,703 | **+36.1%** ⚠️ | 4社 |
| 横浜 | 1,961,481 | 1,936,287 | −1.3% | 5社 |

渋谷の+36%は事業者内の報告基準変更（§10）。`flag_yoy` で検出。

## 6. 指標③：増減率パターン2（コロナ前→後）

「2020–2024で存在する最新データ」/「2011–2019で存在する最新データ」。

### 6.1 ロジック（事業者単位・ガード付き）
各運営会社について:
```
pre_year  = max{ y ∈ 2017..2019 | データ有無==1 かつ 値>0 }
post_year = max{ y ∈ 2023..2024 | データ有無==1 かつ 値>0 }   ← 2022以前は不可
```
両方そろう社だけを basket に:
```
分母 = Σ 値(社, pre_year)        # 除外社は pre も post も使わない（basket一貫）
分子 = Σ 値(社, post_year)
rate_covid = 分子/分母 − 1       # basket が空なら NaN
```

### 6.2 ポイント
- **参照年は社ごとに違ってよい**。各社は自分自身を比較（like-for-like）するので、合算は乗客数で重み付けした平均になる。
- **pre窓(2017-2019)とpost窓(2023-2024)の非対称は意図的**：
  - pre = コロナ前の平常年は互いに代替可能（2018でも基準として妥当）。
  - post = 回復軌道の途中で年ごとに大きく違い、**底値(2020-22)を「後」扱いできない**。
- 参照年は実質 **2019→2024 に自然収束**（pre: 8,118社が2019、post: 7,728社が2024）。欠測時のみ自動で繰り上がる。

### 6.3 実測
- 算出可能: **7,684群**（post厳格化後）。中央値 約 **−8.5%**（コロナ後の未回復を正しく捉える）。

| 例 | pre(年) | post(年) | rate_covid |
|---|---|---|---|
| 東京 | 1,339,555 (2019) | 1,262,604 (2024) | **−5.7%** |
| 新宿 | 3,265,570 (2019) | 2,713,386 (2024) | −16.9% |
| 渋谷 | 2,704,384 (2019) | 2,897,703 (2024) | +7.1% |
| 横浜 | 2,101,709 (2019) | 1,936,287 (2024) | −7.9% |

> 東京は、単純合算なら+37%（新幹線の新規計上による見かけ増）だが、like-for-like で **−5.7%** と正しくなる。

## 7. 品質フラグ

| フラグ | 条件 | 意図 |
|---|---|---|
| `flag_yoy` | \|rate_yoy\| > 30% | 事業者内の報告基準変更（渋谷+36%, 小諸+119%等）を検出 |
| `flag_covid` | 被覆 < 100% **or** いずれかの社の pre_year < 2019 | 一部社のみ / 参照年が非理想 を検出 |
| `level_complete` | 在の社集合が全データ年で一定（bool）| False＝どこかの年で社が抜けた（凹みあり）。詳細表で要確認 |

> post窓を2023/2024に絞ったため、「post非理想」条件は原理的に発生せず、`flag_covid` は上記2条件に簡素化された。

## 8. テーブル設計

**2テーブル構成**で確定。メインは「1群1行のワイド」── 今後 人口・地価などのグループ単位特徴量を右に追記していく feature mart。

### 8.1 メイン `station_group`（1群1行・これを使い続ける）

| 区分 | 列 | 内容 |
|---|---|---|
| 識別 | `grp`, `駅名`, `lon`, `lat` | キー＋地図表示 |
| レベル時系列 | `pax_2011` … `pax_2024`（14列）| inclusive合算・補間なし・全社欠測年はNULL |
| レベル品質 | `n_op` | 群の延べ運営会社数 |
| レベル品質 | `level_complete`（bool）| 構成が全データ年で一定か |
| 比率P1 | `rate_yoy`, `flag_yoy` | 2024/2023 と異常値フラグ |
| 比率P2 | `rate_covid`, `flag_covid` | コロナ前後 と信頼性フラグ |
| 将来追加 | `population_*`, `landprice_*` … | ここに追記していく |

設計判断:
- **各年合算と比率を同居**：どちらもグループ単位なので自然。かつ比率は合算列から導出不可（basketが違う）なので別列で保持が必須。
- `pax_latest` は持たない（`pax_2024` で代用可）。
- 各年の `n_op` を14列持つのは冗長（98.2%が一定）。代わりに `level_complete`（bool）1列で軽くし、詳細は②へ逃がす。

### 8.2 監査用 `station_group_operator`（1群×1社・普段は使わない）

| 列 | 内容 |
|---|---|
| `grp`, `運営会社` | キー |
| `pax_2011` … `pax_2024` | 各社の年次値 |
| `present_2011` … `present_2024` | 在/欠測（データ有無==1か）|
| `pre_year`, `post_year` | P2で実際に使った参照年 |

役割: メインの数字を検証でき、必要時に「各年の寄与社数 `n_op`」「どの社がいつ抜けたか」を再構成できる。メインを汚さず透明性を担保。

## 9. 不採用にした案と理由

検討の過程で却下した代替案を、却下理由（多くは実データ）とともに記録する。

### 9.1 欠測(0/空欄)をそのまま0として全行を単純合算
- **却下**。路線の計上開始/終了で見かけの急増減が出る。例: 東京駅の単純合算は2011→2024で **+37%** だが、新幹線(JR東海)の新規計上による見かけ増で、実態は **+16%**。

### 9.2 構成単位を「路線」にする
- **却下**。事業者内で重複コードにより代表行が年で移動するため路線単位は不安定。**運営会社単位**なら値が保存される。

### 9.3 比率パターン2を「全駅で年を固定」（ユーザー案B相当）
- **却下**。被覆率テーブル上、**全駅が揃う年が存在しない**（毎年1,500〜2,600行が欠測）。2019年の符号変更・2018/19の調査拡大もあり、共通年窓では大半を捨てることになる。

### 9.4 比率パターン2で「欠損社を捨てて年を固定」 vs 「全社揃う年まで遡る」
ユーザーが提起した2案と、採用案Cの比較:

| 案 | 内容 | 判定 |
|---|---|---|
| 案A | 年を2019固定、欠損社は捨てる | **却下**。健全な社を落とし、collapseする社を残して**−100%のゴミ値**を生むことがある（下記 小諸）|
| 案B | 全社揃う年まで遡る | **却下**。1社が古いと全社を古い年へ引きずり、コロナ影響に長期トレンドが混入 |
| **案C** | 社ごとに自分の最新年（混在可）| **採用**。欠損に強く、全社を活かし、古年汚染も最小 |

- **実頻度**：多社群432のうち、群内でpre最新年が食い違うのは**55群のみ**（98%は2019に収束）。案Aと案Cが1pt超ずれるのは**9群**だが、ずれる時は案Aが危険。
- **決定例（小諸駅）**：しなの鉄道(2018→2024で健全に増)とJR東日本(2019のみ, 2024は0)。
  - 案A = JR東を残しJR東2019→2024=0 → **−100%**（駅が消滅した扱い）。実態は元気に増加。
  - 案C = しなの2018→2024 を採用 → **+58.0%**（妥当）。

### 9.5 レベル時系列の内部欠損を線形補間
- **却下（デフォルト）**。線形補間は**コロナを知らない**ため、2019→2023を直線で結ぶと**2020-2022の谷を消して過大評価**しうる。
- **実態**：内部欠損年は **2021:338社, 2022:322社** に集中（コロナ年に欠損が多発）。ただし多くは2020(谷)を実測で持ち欠損は2021-22なので、補間しても回復ランプを描くだけで害は小さい。真に危険な「2020欠損で谷を跨ぐ」ケースは北越急行系など小規模駅に限られる。
- **結論**：害が小さくても捏造を避け、**補間しない**。社が抜けた年は合算が凹むが、`n_op` の低下で「データ穴」と判別できる。「上に盛る」より「下に凹む」方がコロナ期の誤解を生まず安全。
- 折衷案（コロナ外の内部欠損だけ補間）は当面採らない（必要時に再検討）。

### 9.6 メインテーブルをロング(tidy)形式にする
- **却下**。プロットは楽だが、人口・地価（グループ単位1値）を14年分の各行に繰り返す必要が出て非効率・破綻しやすい。**ワイド1枚**なら右に列を足すだけ。ワイドでも melt でグラフ化できる。

### 9.7 メインに各年 `n_op` を14列持つ / `pax_latest` を持つ / `cov_covid`(連続値) を持つ
- いずれも**却下（最小化）**。`n_op`×14は98.2%が一定で冗長→`level_complete`(bool)＋詳細表で代替。`pax_latest`は`pax_2024`で代用。`cov_covid`連続値は`flag_covid`(bool)に畳む（閾値調整したい場合のみ復活）。

## 10. 残存リスク

like-for-like basket でも消せない問題が1つある：**事業者の集計範囲自体が年で変わる**ケース。

- **実例（渋谷・東急東横線）**: 2023=1,036,352 → 2024=**1,770,430（+71%）**。1年の実需増ではなく、東急の渋谷集計の定義変更（直通・複数線の合算化）と見られる。事業者「内」の変化なので basket を揃えても検出できない。
- **対策**: 算出後の異常値フラグ `flag_yoy`（\|率\|>30%）で自動採用を防ぎ、可視化・分析で除外/注記する。パターン2は基準が2019(高水準)のため影響が薄まる（渋谷P2は+7.1%）。

## 11. リファレンス実装（擬似コード）

```python
# 前処理 ──────────────────────────────────────────────
# 乗降客数を数値化。代表点→駅名ごとに haversine DBSCAN(eps=1km) で grp 付与。
# (grp, 運営会社, 年) に集約: present=(データ有無==1).any(), pax=乗降客数.sum()  # 重複行は0

YEARS = range(2011, 2025)
PRE_W, POST_W = [2017,2018,2019], [2023,2024]

# 指標①レベル時系列（inclusive・補間なし）──────────────
for y in YEARS:
    present_ops = {社 | present(grp,社,y)}
    pax_[y]  = Σ pax(社,y) for 社 in present_ops   # 全社欠測なら NULL
    n_op_[y] = len(present_ops)
level_complete = (在の社集合が全データ年で一定)

# 指標②パターン1（2024/2023）─────────────────────────
basket1 = {社 | present(2023) & present(2024) & pax(2023)>0}
rate_yoy  = Σpax2024(basket1)/Σpax2023(basket1) - 1
flag_yoy  = abs(rate_yoy) > 0.30

# 指標③パターン2（コロナ前→後・事業者単位）──────────────
for 社 in 群:
    pre_year[社]  = max{ y∈PRE_W  | present(社,y) & pax(社,y)>0 }
    post_year[社] = max{ y∈POST_W | present(社,y) & pax(社,y)>0 }
basket2 = {社 | pre_year[社] and post_year[社]}        # 両方ある社のみ（pre/post一貫）
rate_covid = Σpax(社,post_year)/Σpax(社,pre_year) - 1  # 空なら NaN
flag_covid = (len(basket2)/n_op < 1) or (min(pre_year[basket2]) < 2019)
```

実装は `script/create_dataset_for_AI_Database_Map.ipynb` に、既存ファイルを壊さない形で追加する。
