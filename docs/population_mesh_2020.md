# 国勢調査 人口及び世帯メッシュ（2020）

駅からの半径内人口を算出するために取得した、2020年（令和2年）国勢調査の**地域メッシュ人口・世帯データ**の概要・取得方法・懸念点・集計設計をまとめる。

**500m と 250m の2解像度を取得**し、**半径集計には 250m メッシュを全半径で採用**（決定B。理由は §7）。500m は参照・フォールバック用に保持。関連: [`CRS.md`](./CRS.md), [`passenger_aggregation.md`](./passenger_aggregation.md)。

---

## 1. 概要

| 項目 | 内容 |
|---|---|
| 出典 | 総務省 e-Stat「令和2年国勢調査 世界測地系メッシュ」地域メッシュ統計 |
| 統計コード | `00200521`（国勢調査）/ searchKind=2（地域メッシュ）|
| 取得解像度 | **250m（5次メッシュ・10桁）＝採用** ／ 500m（4次・9桁）＝保持 |
| 取得項目 | **人口総数（cat01=0010）** と **世帯総数（cat01=0340）** |
| 範囲 | **日本全国**（第1次地域区画ごと **各151区画＝全国完備**）|
| 取得日 | 2026-06-23（e-Stat API v3.0, `getSimpleStatsData`）|

提供単位は **「第1次地域区画」**（県境でメッシュ値を分割せず全人口を保持）。半径集計で県境をまたいで全員数えるのに適する。

## 3. 取得方法（再現手順）

e-Stat API の appId は `.env` の `ESTAT_APP_ID`（gitignore済み）から読み込む。**appId・URLはコード出力に出さない**。

1. **対象表の特定**: `getStatsList`（statsCode=`00200521`, searchKind=`2`, surveyYears=`2020`）→ `STATISTICS_NAME` に `250M`（または `500M`）を含む表＝**各151表**（1次メッシュ区画ごと、`statsDataId`）。
2. **データ取得**: 区画ごとに `getSimpleStatsData`（CSV）、`cdCat01=0010,0340`（人口・世帯）。
   - 250mは人口＋世帯で最大 約11.5万件（M5339等）と**10万件上限を超える区画が2つ**あるため、**`startPosition` ページング**で全件取得（残りページのデータ行のみ追記）。他149区画は1コール。
   - `metaGetFlg=Y`, `sectionHeaderFlg=1`。
3. 区画別に `mesh250_<region>.csv` として保存、`_manifest.csv`（region/statsDataId/records/file）を生成。**再開可能**（取得済みskip）・3回リトライ。

> 全50項目（年齢別・世帯種類別）が必要なら同手順で `cdCat01` を拡張（250m全項目は全国 約5,819万レコード）。

## 6. 懸念点・注意

### 6.1 秘匿・合算（cat02）— 250mは500mより大きい
- 250mは**セルの48.6%が秘匿/合算**、ただし**人口ベースでは8.0%**（500mは4.3%）。残り**92%の人口はクリーン（無し）セル**。
- 合計は正確だが、フラグセルの値は隣接へ集約され**所在が数百mずれうる**。半径が小さいほど影響。
- 対策: 全セルをそのまま面積按分（総量正確）。任意で `pop_500m_hidden_ratio`（最小半径内の秘匿/合算割合）を品質フラグ化。

### 6.2 一様密度仮定 ← 250m採用の主目的
- 各メッシュ内は人口一様とみなす。**250mは500mに対しセル面積1/4で、最小半径(500m)の一様密度誤差を大幅に低減**。これが250m採用の主理由（§7）。

### 6.3 測地系（JGD2000/JGD2011）
- API統計値は「世界測地系」で**測地系非依存**。メッシュ幾何はコードから生成し **EPSG:6668(JGD2011)** 扱い（[`CRS.md`](./CRS.md)）。

### 6.4 境界ファイルは不要
- メッシュ幾何は**コードから一意に計算可能**（§8.2、東京駅で検証済）。境界shpは冗長で精度も上がらない（e-Stat標準メッシュ境界はフル矩形＝海陸クリップなし）。沿岸の海側按分を厳密化したい場合のみ、N03で陸地クリップする改良が有効（任意・効果限定）。

### 6.5 取得項目・計算量・その他
- 取得は **人口総数・世帯総数のみ**（半径用途に十分。年齢別等は §3 で追加可）。
- 250mは約116万セル。半径集計は **KDTree＋内部/境界分割**で現実的（§8.4）。大半径バッファは駅間で大きく重複（加算不可）。
- 将来推計人口（国土数値情報, 1km/500m/250m）は別データ。**250mで揃えれば現在・将来を同一解像度**で扱える（将来推計250mの全国網羅性は取得時に要確認）。

## 7. 採用方針：250mメッシュ単一ソースで全半径（決定B）

### 7.1 決め手：同一駅・同一半径での両メッシュ推計の差
サンプル約200駅（主要駅＋ランダム）で、500m/250m両メッシュの**面積按分人口**を実測比較した結果:

| 半径 | (250m−500m)/500m 中央値 | 絶対差 中央値 | 絶対差 90%tile | メッシュ選択の影響 |
|---|---|---|---|---|
| **500m** | +3.2% | **6.4%** | **24.2%** | **大きい** |
| **1km** | +0.7% | 2.2% | 6.4% | やや |
| **2km** | +0.2% | 0.8% | 2.7% | ほぼ無し |
| 5km以上 | ≈0 | <1%（収束）| — | 無し |

→ **メッシュ解像度が効くのは 500m半径（と多少1km）だけ。2km以上は両者ほぼ一致**（＝大半径なら500mでも同等）。

### 7.2 250mが小半径で優れる理由
- 250mはセル面積1/4で**一様密度誤差が小さい**。駅は密度の高い場所にあり、近傍の人口集中を細かく捉える。
- 500m半径で中央 +3.2% 高いのは、500mメッシュが駅近傍の密度を平準化して**過小評価**しがちなのを250mが拾うため。**250mがより真値に近い**と考えられる。

### 7.3 秘匿増は実害なし
- 半径内の秘匿/合算が占める人口割合は250mで増える（500m半径で約1.8%、2kmで約6.5%。500mメッシュは各0.0%/2.0%）が、**全国総量は1人単位で公式一致**＝再配分のみで欠落なし。
- 秘匿が多い2km以上で両メッシュ推計が一致（<1%）＝**再配分は半径内で相殺され推計を歪めない**。500m半径でも「秘匿exposure(1.8%) < 解像度改善(6.4%)」で純益はプラス。
- 参考（東京M5339のセルレベル）: 250m=57,605セル(平均525人) vs 500m=16,671セル(平均1,812人)、地域合計は完全一致（総量保存）。

### 7.4 結論
- **250mを全半径で採用（単一ソース）**。500m・1km半径で明確に優れ、2km以上は同等、秘匿増は無害、`pop_500m ≤ pop_1km ≤ … ≤ pop_20km` の**単調性も保証**。
- **メッシュ混在（半径ごとに解像度を変える）は不採用**（ソース差で稀に単調性が崩れる）。
- 唯一の代償は計算量・容量（500mの約2.5倍）。§8.4 の KDTree＋内部/境界分割で処理可能。
- （もし要件が「2km以上のみ」なら500mで同等・軽量だが、**500m/1km半径が対象**のため250mが勝る。）

## 8. 駅半径内人口 面積按分ロジック（詳細設計）

各駅グループ（`data/derived/station_group.csv` の `lon`/`lat`）に対し、**250mメッシュ**で
半径 **500m / 1km / 2km / 5km / 10km / 20km** の人口を**面積按分**で算出する。
（過去実装 `script/open_data.ipynb` は1kmメッシュ境界shp＋全overlayで1/2/5/10kmを算出。本設計は250mメッシュ・コード生成・内部/境界分割で 500m/20km まで拡張。）

### 8.1 入力・出力
- 入力: `station_group`（9,273群, EPSG:6668）／ 250mメッシュ（1,163,757セル, 人口・世帯）。
- 出力: `station_group` に **`pop_500m, pop_1km, pop_2km, pop_5km, pop_10km, pop_20km`**（整数）を追加。任意で **`setai_*`**（世帯）と `pop_500m_hidden_ratio`（品質フラグ）。

### 8.2 メッシュ → ポリゴン（コードから生成。境界shp不要）
250mメッシュの**10桁コード**を緯度経度矩形へ変換（標準地域メッシュ JIS X 0410。検証済み）:

```
code = aa bb c d e f g h   (10桁)
  1次: aa=floor(lat*1.5), bb=floor(lon)-100
  2次: c(0-7)緯, d(0-7)経                    （1次を8×8）
  3次(1km): e(0-9)緯, f(0-9)経                （2次を10×10）
  4次(500m): g(1-4)  1=SW,2=SE,3=NW,4=NE      （1kmを2×2）
  5次(250m): h(1-4)  同上                      （500mを2×2）
南西角: lat_s = aa*2/3 + c*(2/3)/8 + e*(2/3)/80 + ((g-1)//2)*(1/240) + ((h-1)//2)*(1/480)
        lon_w = (bb+100) + d/8 + f/80          + ((g-1)%2)*(1/160)  + ((h-1)%2)*(1/320)
セル幅: Δlat = 1/480°, Δlon = 1/320°   （≈ 250m四方）
```
検証: `5339461132` → lon[139.7656,139.7688]×lat[35.6792,35.6813] で**東京駅を内包** ✓。
（500mは9桁＝末尾 `h` を除いた式。`jismesh` 等でも可。）
各セルに人口(cat01=10)・世帯(cat01=340)を紐付け、**全フラグ(無し/合算/秘匿)の値をそのまま使用**（§5で全国合計が公式と一致＝二重計上なしを実証済）。

### 8.3 投影（CRS）
- 結合・格納は **EPSG:6668**。距離・面積・バッファ計算時のみ [`CRS.md`](./CRS.md) の**正積図法（Albers, GRS80）**へ `to_crs`。
- バッファ円は正積投影上で `point.buffer(R[m])`。面積比は正積で厳密。半径ひずみ <~1.5%（メッシュ250m粒度に対し無視可）。

### 8.4 集計アルゴリズム（内部/境界分割で厳密かつ高速）
全セルを正積座標へ投影し、**重心**で `scipy.spatial.cKDTree` を1回構築。

判定用マージン `h` ＝ **正積投影後のセル半対角の上限**。理論値は √2×125≈177m だが、正積図法のひずみで実際は
緯度依存で **約169m（北海道）〜195m（沖縄）**（実測）。**取りこぼし・内部誤判定を防ぐため安全側に `h = 200m`**
とする（候補を多めに拾うのは無害＝外側セルは交差面積0でweight0）。

各駅 × 各半径 R:
1. KDTreeで **重心距離 ≤ R+h** の候補セルを取得。
2. 3分類:
   - **内部**（距離 ≤ R−h、完全に円内）→ `weight = 1`（和のみ）。
   - **境界**（R−h < 距離 < R+h）→ `weight = area(mesh ∩ buffer)/area(mesh)`（shapely交差）。
   - 外部 → 除外。
3. `pop_R = Σ pop_cell × weight`（世帯も同様）。

**バッファ生成**: `point.buffer(R, quad_segs=32)` 以上を使う。既定（quad_segs=8）は多角形近似で円面積を
約0.6%過小評価し（実測 99.36%）、境界比率の高い小半径で pop を下振れさせるため。quad_segs=32 で 99.96%。

**要点**: ポリゴン交差は**境界セルのみ**。内部は単純和。これで20km半径まで現実的時間で厳密按分（250mは約116万セル、内部和はKDTree範囲、境界交差は全駅×全半径で概ね1千万回規模＝数分〜十数分）。`h=200m`は内部・候補双方の安全マージンを兼ねる。

境界セル寄与比率 ≈ 0.25/R(km)（250mはセルが細かく境界比が500mの半分）:
| 半径 | 推奨手法 |
|---|---|
| 500m / 1km / 2km | 面積按分（必須〜推奨）|
| 5km / 10km / 20km | 面積按分（重心法でも誤差数%・高速化したい場合の代替可）|

### 8.5 整合性・エッジケース
- **単調性**: **全半径を同一手法（面積按分）**にすれば `pop_500m ≤ pop_1km ≤ … ≤ pop_20km` は厳密成立（入れ子のバッファで各セルのweightが単調増加するため）。§8.4で大半径に重心法を併用すると厳密保証は崩れうるので、その場合は `pop_R = max(pop_R, pop_{直前半径})` でクランプするか、手法を統一する。違反検知をテストに入れる。
- **海上・無居住**: 人口のないメッシュは不在＝寄与0（沿岸駅の大半径が低いのは実態通り）。
- **駅間重複**: 大半径は駅間で重複。各駅独立指標であり加算しない。
- **端数**: 按分でfloat→最後に整数丸め（`Int64`）。

### 8.6 実装手順（要約）
1. 151区画CSV（mesh250）のデータ部（各ファイルの最初の `"VALUE"` 行以降の全データ行。ページング追記分も含む）を結合 → `cat01==10`(人口)・`cat01==340`(世帯)を抽出。各セルは (area, cat01) で一意（重複なし＝§5で総量一致を確認済）。
2. 10桁コードから矩形ポリゴン＋重心を生成（EPSG:6668）→ 正積投影。
3. 重心で cKDTree 構築。
4. `station_group` 代表点を正積投影。各駅×各半径で §8.4 を実行。
5. `pop_500m … pop_20km`（＋任意で世帯・品質）を `station_group` に結合、CSV更新。
6. 単調性・既知駅（東京・新宿等）でスポット検証。
